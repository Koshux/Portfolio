# Portfolio вҖ” Agent Operating Manual

This file is the single source of truth for AI coding agents working in this repo.
It is intentionally compatible with both **GitHub Copilot** (`AGENTS.md` is supported
as workspace instructions) and **Claude Code** (which reads `AGENTS.md`/`CLAUDE.md`
from the project root). Do **not** create a separate `copilot-instructions.md` or
`CLAUDE.md` вҖ” link to or extend this file instead.

> Project: `jameslanzon.com` portfolio. Static site generated by **Nuxt 4** and
> deployed to **GitHub Pages**. See [nuxt.config.ts](nuxt.config.ts).

---

## 1. Hard rules (do not break)

1. **Never run dev servers in the foreground.** Use `npm run dev` only when the
   user explicitly asks. Prefer `npm run generate` + static preview for
   verification.
2. **Static-only output.** The site is built with `nitro.preset = 'github-pages'`
   and `static: true`. Do not introduce server routes, API handlers, or runtime
   middleware that require a Node server.
3. **No asset paths starting with `_`** (GitHub Pages strips them). Keep
   `app.buildAssetsDir = 'assets'` in [nuxt.config.ts](nuxt.config.ts).
4. **Every change goes through the SDLC** described in В§3. No "drive-by"
   features without a journey + spec + log entry.
5. **Tests are required** for any new component, composable, or content schema.
   See В§5.
6. **Never delete files in [docs/logs/](docs/logs/)** вҖ” they are the audit trail.
7. **Do not commit secrets.** There is no backend; if one is ever added, secrets
   live in GitHub Actions only.

## 2. Project structure

```

public/               # Static passthrough (robots.txt, favicons, og images)

docs/
  journeys/           # User journeys (the "why")
  specs/              # Technical specs (the "what")
  logs/               # Implementation logs (the "what was done")
  decisions/          # ADRs (the "why we chose X")
tests/
  unit/               # Vitest unit tests
  integration/        # Vitest + @nuxt/test-utils integration
  e2e/                # Playwright end-to-end
.github/
  instructions/       # File-scoped agent guidance
  prompts/            # SDLC slash-commands (/journey, /spec, /implement, /log, /test)
  agents/             # Specialist subagents

  workflows/          # GitHub Actions CI
```

The legacy `website/` and root `index.html` exist only because the site is
mid-migration. Treat them as read-only; new work belongs under `app/`.

## 3. The SDLC for AI agents

Every meaningful change follows these five phases. Each phase has a slash
command (`/journey`, `/spec`, `/implement`, `/log`, `/test`) and a matching
template under [docs/](docs/). See
[docs/sdlc/README.md](docs/sdlc/README.md) for the full playbook.

```
в”Ӯ Journey  в”Ӯ вҶ’ в”Ӯ Spec в”Ӯ вҶ’ в”Ӯ Implement  в”Ӯ вҶ’ в”Ӯ Test в”Ӯ вҶ’ в”Ӯ Log  в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

```

| Phase | Output | Template |
|---|---|---|
| Journey | `docs/journeys/JNY-XXX-*.md` | [template](docs/journeys/_TEMPLATE.md) |
| Spec | `docs/specs/SPEC-XXX-*.md` | [template](docs/specs/_TEMPLATE.md) |
| Implement | code under `app/` | follow spec acceptance criteria |
| Test | tests under `tests/**` | see В§5 |
| Log | `docs/logs/YYYY-MM-DD-*.md` | [template](docs/logs/_TEMPLATE.md) |

IDs are assigned by the next available number (`JNY-001`, `SPEC-001`, вҖҰ).
The `/log` command is mandatory after every implementation session вҖ” it is the
only mechanism we have to recover context across agent runs.

## 5. Testing strategy

Four layers, each with a clear purpose. Do not collapse them.

| Layer | Tool | Lives in | Use for |
|---|---|---|---|
| Unit | Vitest | `tests/unit/` | Pure functions, composables, utils |
| Component | Vitest + `@vue/test-utils` | `tests/unit/components/` | Vue components in isolation |
| Integration | Vitest + `@nuxt/test-utils` | `tests/integration/` | Nuxt rendering, content queries, routing |
| E2E / browser | Playwright | `tests/e2e/` | Real browser, generated static output |

**Browser exploration vs E2E**: Use Playwright for *committed* repeatable tests.
Use the **Chrome DevTools MCP** tools (`mcp_io_github_chr_*`) only for
exploratory debugging вҖ” never as a substitute for a Playwright test. See
[.github/instructions/testing.instructions.md](.github/instructions/testing.instructions.md).

**Coverage targets** (enforced in CI):
- Unit + component: вүҘ 80 % line coverage of `app/composables` and `app/components`
- E2E: every public route has at least one smoke test

## 6. Code conventions

- **Vue 3 `<script setup lang="ts">`** with the Composition API. No Options API.
- **Tailwind** for styling вҖ” no scoped CSS unless Tailwind cannot express it.
- **Composables** in `app/composables/` named `useXxx.ts`, one composable per
  file, return a plain object.
- **Content** in `content/` is the source of truth for copy. Components render
  it via `<ContentRenderer>` or `queryCollection()`. Do not hard-code copy in
  `.vue` files.
- **Images** go through `<NuxtImg>` so `@nuxt/image` can optimise them.
- **Fonts** via `@nuxt/fonts` only вҖ” no `<link rel="stylesheet">` to Google
  Fonts.
- **Imports**: rely on Nuxt auto-imports. Only add explicit imports when an
  auto-import collision occurs.

## 7. Agent behaviour expectations

- **Read before you write.** Always open the relevant spec under
  [docs/specs/](docs/specs/) before editing code.
- **One task, one log.** Append a log entry under [docs/logs/](docs/logs/) at
  the end of every session вҖ” even if the work was abandoned. Failed attempts
  are valuable.
- **Use the specialist subagents** for their domain. See
  [.github/agents/](.github/agents/). Don't reinvent their workflows in the
  parent agent.
- **Prefer the orchestrator skill** (`/sdlc-workflow`) when starting a new
  feature from scratch вҖ” it walks through every phase with the right templates.
- **Confirm before destructive ops** (deleting files, `git reset`, force-push,
  rewriting history).
