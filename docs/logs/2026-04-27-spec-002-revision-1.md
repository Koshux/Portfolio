**Minimum-legal rationale.** GDPR Art. 7(3) requires withdrawing consent to
be as easy as giving it; Art. 13 requires that withdrawal information be
provided at the point of collection. ePrivacy Art. 5(3) makes consent the
lawful basis for non-essential storage/access on the device. None of these
mandate a sitewide footer link — they require an accessible, persistent
withdrawal path. The header contact menu satisfies "as easy and persistent"
without the visual surface of a footer the rest of the site does not need.
Hosting the trigger on `/legal/privacy` (where the disclosure already lives)
keeps the withdrawal mechanism at the point of explanation.

### Issue 3 — `[object Object]` bug

- **Root cause.** `@nuxt/content` v3 collections of `type: 'page'` always
  expose the parsed markdown AST under a top-level `body` property on the
  document. The consent collection's frontmatter also defined a `body:`
  string field; on hydration the AST shadowed the frontmatter string, so
  `{{ promptCopy.body }}` rendered an object → `[object Object]`.
- **Fix.** Renamed the frontmatter field `body` → `prompt` everywhere it
  flows: [content/legal/consent.md](../../content/legal/consent.md),
  [shared/content-schemas.ts](../../shared/content-schemas.ts) (with an
  inline comment documenting the shadow),
  [app/components/Ui/ConsentPrompt.vue](../../app/components/Ui/ConsentPrompt.vue)
  (`Props.prompt: string` and `{{ prompt }}`), and the layout binding in
  [app/layouts/default.vue](../../app/layouts/default.vue)
  (`promptCopy.prompt` + `:prompt="promptCopy.prompt"`).

## Issues encountered & resolution

- **Symptom**: Consent prompt rendered `[object Object]`.
  **Diagnosis**: `@nuxt/content` v3 `type: 'page'` documents always carry
  the parsed markdown AST under a top-level `body` property. The
  frontmatter's `body:` field was shadowed by the AST.
  **Fix**: Renamed the frontmatter field to `prompt` across content,
  schema, component props, and layout binding. Added a comment in the
  schema so the next agent does not re-introduce the collision.

- **Symptom**: `tests/integration/pages/legal-privacy.spec.ts` H1
  assertion failed even though the H1 was rendered.
  **Diagnosis**: Vue SSR injects fragment markers (`<!--[-->` /
  `<!--]-->`) inside element text on hydration boundaries, which broke
  the strict `<h1>Privacy notice</h1>` substring match. The `not.toContain('Cookie preferences')` assertion also failed because the markdown
  body itself contains the words "Cookie preferences".
  **Fix**: Use a tolerant regex tolerant of comment markers for the H1
  check, and assert absence by `data-testid` rather than substring.

- **Symptom**: 8 mobile-project Playwright failures —
  `header a[href="/legal/privacy"]` reported `expect(toBeVisible)` failed.
  **Diagnosis**: At the 360×600 mobile viewport the inline desktop link
  is `display:none` (Tailwind `hidden md:flex`) and the dropdown link is
  hidden inside a closed `<details>`.
  **Fix**: Added a `clickHeaderPrivacyLink(page)` helper that prefers the
  inline desktop link when visible and otherwise opens the dropdown
  before clicking. Smoke test now asserts presence of both surfaces in
  the DOM rather than depending on viewport-specific visibility.

## Hand-off notes (for the next agent)

- Do **not** reintroduce a `<footer>`. The minimum-legal placement is
  recorded in SPEC-002 AC-9 and AC-26; the integration and e2e suites
  will fail loudly if a footer reappears.
- The Privacy link in the header contact menu is **always** rendered, even
  in inert builds. The Cookie preferences trigger is rendered **only** on
  `/legal/privacy` and **only** when `NUXT_PUBLIC_GA_MEASUREMENT_ID` is
  set.
- Re-opening the consent prompt across routes goes through
  `useConsent().requestReopen()` (a useState-backed monotonic counter the
  layout watches). Do not bypass it with direct DOM events.
- `@nuxt/content` v3: never name a frontmatter field `body` on a
  `type: 'page'` collection — it will be shadowed by the parsed AST.
