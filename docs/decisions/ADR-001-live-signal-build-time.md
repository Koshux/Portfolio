# ADR-001 — Resolve the live-signal chip data at build time

## Context

`jameslanzon.com` is a static site generated by Nuxt 4
(`nitro.preset = 'github-pages'`, `static: true`) and deployed to GitHub
Pages. [JNY-001](../journeys/JNY-001-portfolio-revamp-iteration-1.md) calls
for a small "live signal" in the persistent header: the timestamp of James'
most recent public commit across his GitHub repositories, plus the current
time in Malta. The chip is informational — it telegraphs "this is a real,
active engineer" — and must satisfy the journey's hard constraints:

- The site must remain fully readable, including the chip's commit data,
  with **JavaScript disabled** (per JNY-001 success criteria and SPEC-001
  AC-14).
- There is **no backend**. We cannot proxy GitHub through our own server,
  and we will not introduce one in iteration 1.
- A failed or slow upstream call must **never** break the build, the page
  render, or the visitor's experience.
- Visitor browsers must not consume our GitHub API rate limit budget — the
  site receives unauthenticated traffic and the public unauthenticated
  limit is 60 requests/hour per IP.

The Malta clock is a separate concern (progressively enhanced via
`useMaltaClock()`); this ADR is only about the commit-data half of the
chip.

## Decision

Resolve the live-signal commit data **at build time**.

A Node 20 ESM script,
[scripts/fetch-live-signal.mjs](../../scripts/fetch-live-signal.mjs), runs
during the `prebuild` and `pregenerate` npm lifecycle hooks. It calls
`https://api.github.com/users/jameslanzon/events/public?per_page=30`,
extracts the most recent `PushEvent`'s `repo`, short `sha`, and ISO
`timestamp`, validates the result against the `liveSignal` Zod schema in
[content.config.ts](../../content.config.ts), and writes
`content/live-signal.json`. On any failure mode (non-200 response, HTTP
403 + `X-RateLimit-Remaining: 0`, network error, malformed JSON, no
`PushEvent` in the page) the script writes the discriminated-union
fallback `{ unavailable: true, fetchedAt: <ISO> }` and exits `0`. The
build never fails because of this script.

`content/live-signal.json` is **committed and tracked** with a placeholder
fallback so `nuxt prepare` (run via `postinstall`) succeeds on a fresh
clone before any build hook has fired. Local clones run
`node scripts/setup-dev.mjs` once after `npm install`; it applies
`git update-index --skip-worktree content/live-signal.json` so subsequent
build-time writes don't appear in `git status` and never get committed
back. CI checkouts are ephemeral, so this is a no-op there.

The script also honours `process.env.SKIP_LIVE_SIGNAL_FETCH=1` (early
exit, no read or write — used by integration tests that pre-seed the
JSON) and `process.env.GITHUB_TOKEN` (sent as `Authorization: Bearer …`
to lift the unauthenticated rate limit).

## Alternatives considered

| Option | Pros | Cons | Why rejected |
|---|---|---|---|
| **A. Client-side fetch** from the visitor's browser to `api.github.com`. | Always-fresh data; no build coupling. | Breaks the JS-disabled requirement (AC-14); leaks our rate-limit budget to anonymous visitors (60/h per IP — trivially exhausted from a single coffee-shop NAT); CORS works for `api.github.com` but introduces a runtime dependency the site otherwise has none of; visitor latency cost. | Violates JNY-001 success criteria and AC-14. |
| **B. Scheduled GitHub Action** that re-runs the fetch on a cron and opens a PR (or pushes to a deploy branch) when the JSON changes. | Decouples freshness from James' deploy cadence; chip stays current even between content updates. | Significantly more CI ceremony (separate workflow, bot identity, PR auto-merge, branch protection interactions) for marginal value in iteration 1; adds a second source of truth for `content/live-signal.json`; conflicts with the skip-worktree pattern. | Too heavy for iteration 1. **Parked as iteration-2 candidate** — recorded in SPEC-001 §Next iteration candidates. |
| **C. Build-time fetch (chosen).** | Zero runtime cost; works with JS disabled; rate limit consumed once per deploy by a single CI runner; never blocks the build (graceful fallback). | Data freshness bounded by deploy cadence — "3 days ago" can become "3 weeks ago" if James doesn't redeploy; build needs network access to GitHub. | — |

## Consequences

### Positive
- Site renders the chip with real commit data using **zero** client JS
  beyond the unrelated Malta clock `setInterval`.
- AC-14 (JS-disabled) satisfied trivially — the chip is server-rendered.
- Visitors do not consume our GitHub rate-limit budget.
- Build never fails because of upstream GitHub issues — the fallback
  branch is always a valid `liveSignal` document.

### Negative
- **Rate limit risk.** Unauthenticated GitHub API limit is 60/hour per IP.
  CI may share an IP with other repos and trip 403 + `X-RateLimit-Remaining: 0`.
  Mitigation: optional `GITHUB_TOKEN` raises this to 5 000/hour. When
  rate-limited, the script writes the unavailable fallback and the chip
  degrades to the static "GitHub · recent activity" string.
- **Data freshness bounded by deploy cadence.** If James does not
  redeploy for a month, the chip will say "1 month ago". Acceptable for
  iteration 1; the relative-time helper keeps it legible. Iteration 2
  may add the scheduled-Action approach (Option B) if this becomes a
  real problem.
- **Minor CI complexity.** A `prebuild` and `pregenerate` hook now run a
  Node script that performs network I/O. Tests cover all five failure
  branches plus the `SKIP_LIVE_SIGNAL_FETCH=1` early-exit, so the hook
  is exercised independently of the rest of the build.

### Neutral
- `content/live-signal.json` is tracked with a placeholder. Local clones
  apply skip-worktree once via `scripts/setup-dev.mjs` to keep
  `git status` clean after local builds. CI is ephemeral and this is a
  no-op there.

## Security

- **CORS / browser exposure.** Not applicable. The fetch script runs in
  Node at build time only. Visitor browsers never call `api.github.com`.
  The deployed static HTML contains only the baked-in `repo`, short `sha`
  and ISO `timestamp` strings (or the unavailable fallback).
- **XSS surface.** All rendered values flow through Vue text
  interpolation (`{{ }}`), which auto-escapes. **`v-html` is not used
  anywhere in this iteration.** `repo` and `sha` come from an external
  API and are still treated as untrusted strings — they are rendered
  only via interpolation, never as HTML.
- **Commit messages.** Deliberately **not** captured by the fetch
  script. Omitting them shrinks the attack surface (no risk of leaking
  in-progress or sensitive commit copy into a public site) and reduces
  the schema/UX surface to the minimum the chip needs.
- **Secrets.** No secrets are required for the site to build. An
  optional `GITHUB_TOKEN` may be supplied via `process.env` (typically a
  GitHub Actions secret in CI). The token is read from the environment
  only — it is never logged, never written to disk, and never appears in
  the generated HTML or in `content/live-signal.json`.
- **Input validation.** The fetched payload is validated against the
  `liveSignal` Zod schema before being written. A schema mismatch lands
  in the unavailable-fallback branch.

## Privacy

- The script consults a single endpoint:
  `https://api.github.com/users/jameslanzon/events/public`. This
  endpoint exposes only public activity that GitHub already publishes
  for any signed-out viewer of `https://github.com/jameslanzon`.
- The fields persisted to `content/live-signal.json` and rendered to the
  static HTML are: `repo` (e.g. `jameslanzon/Portfolio`), `sha` (7-char
  short SHA), `timestamp` (ISO datetime), and `fetchedAt` (ISO datetime
  the script ran). No private repository data, no commit messages, no
  email addresses, and no personally identifying information beyond
  what GitHub already exposes publicly is captured.
- **Visitor data.** Nothing about the visitor leaves their browser. The
  chip is a passive render of pre-baked strings plus the
  `useMaltaClock()` `setInterval`, which is local to the visitor's
  browser and performs no network I/O.

## Follow-ups
